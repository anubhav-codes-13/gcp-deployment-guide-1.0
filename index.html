Step-by-step deployment
1
Setup
Install gcloud CLI + create project
Install the Google Cloud CLI, authenticate, create a new project, and enable all required APIs in one shot.
bash 
# Install gcloud CLI → https://cloud.google.com/sdk/docs/install

gcloud auth login
gcloud projects create ecomm-support-prod --name="Ecomm Support"
gcloud config set project ecomm-support-prod

# Enable required APIs
gcloud services enable \
  run.googleapis.com \
  artifactregistry.googleapis.com \
  secretmanager.googleapis.com \
  storage.googleapis.com \
  cloudbuild.googleapis.com

# Create Artifact Registry (stores Docker images)
gcloud artifacts repositories create ecomm-support \
  --repository-format=docker \
  --location=us-central1
2
Secret Manager
Store API keys safely
Never put secrets in env files or Docker images. Store GOOGLE_API_KEY in Secret Manager and grant Cloud Run access.
bash 
# Store API key (replace with your actual key)
echo -n "YOUR_ACTUAL_GOOGLE_API_KEY" | \
  gcloud secrets create GOOGLE_API_KEY --data-file=-

# Grant Cloud Run service account access
gcloud secrets add-iam-policy-binding GOOGLE_API_KEY \
  --member="serviceAccount:PROJECT_NUMBER-compute@developer.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"
3
Cloud Storage
Upload FAISS indexes + SQLite DB
FAISS indexes and the SQLite database are local files that must live in Cloud Storage so Cloud Run containers can access them at startup.
bash 
# Create storage bucket
gsutil mb -l us-central1 gs://ecomm-support-data

# Upload FAISS indexes and SQLite DB
gsutil -m cp -r backend/data/* gs://ecomm-support-data/data/
gsutil cp backend/ecommerce.db gs://ecomm-support-data/
4
Docker
Build & push Docker images
Build both images locally and push them to Artifact Registry. Run from the project root.
bash 
# Authenticate Docker with Artifact Registry
gcloud auth configure-docker us-central1-docker.pkg.dev

# Build + push backend
docker build -t us-central1-docker.pkg.dev/ecomm-support-prod/ecomm-support/backend:latest ./backend
docker push us-central1-docker.pkg.dev/ecomm-support-prod/ecomm-support/backend:latest

# Build + push frontend
docker build -t us-central1-docker.pkg.dev/ecomm-support-prod/ecomm-support/frontend:latest ./frontend-chat
docker push us-central1-docker.pkg.dev/ecomm-support-prod/ecomm-support/frontend:latest
5
Cloud Run
Deploy both services
Deploy backend first, grab its URL, then inject it as an env var into the frontend deployment.
bash 
# Deploy backend
gcloud run deploy ecomm-backend \
  --image us-central1-docker.pkg.dev/ecomm-support-prod/ecomm-support/backend:latest \
  --platform managed --region us-central1 \
  --allow-unauthenticated \
  --memory 2Gi --cpu 2 --timeout 120 \
  --set-secrets GOOGLE_API_KEY=GOOGLE_API_KEY:latest \
  --set-env-vars BOT_DOMAIN=ecommerce

# Capture backend URL automatically
BACKEND_URL=$(gcloud run services describe ecomm-backend \
  --region us-central1 --format 'value(status.url)')

# Deploy frontend with backend URL injected
gcloud run deploy ecomm-frontend \
  --image us-central1-docker.pkg.dev/ecomm-support-prod/ecomm-support/frontend:latest \
  --platform managed --region us-central1 \
  --allow-unauthenticated \
  --memory 512Mi \
  --set-env-vars NEXT_PUBLIC_API_URL=$BACKEND_URL

Your public URL — how it works
Frontend (share this)
https://ecomm-frontend-abc123-uc.a.run.app
Anyone in the world can open this in any browser
Backend (internal)
https://ecomm-backend-abc123-uc.a.run.app
Called by frontend automatically — users never see this
--allow-unauthenticated
This single flag is what makes it public. Without it → only you (logged into GCP) can access it. With it → anyone with the URL can use it.
Get your live URLs after deploying
bash 
# Frontend public URL
gcloud run services describe ecomm-frontend \
  --region us-central1 --format 'value(status.url)'

# Backend public URL
gcloud run services describe ecomm-backend \
  --region us-central1 --format 'value(status.url)'
⚠️
Important — deploy in this exact order:
Deploy backend first → get its URL → bake it into frontend Docker build → deploy frontend. NEXT_PUBLIC_API_URL is embedded at build time in Next.js, not at runtime.
Correct build order with URL baked in
bash 
# 1. Deploy backend first
gcloud run deploy ecomm-backend ...

# 2. Capture backend URL
BACKEND_URL=$(gcloud run services describe ecomm-backend \
  --region us-central1 --format 'value(status.url)')

# 3. Build frontend with real backend URL baked in
docker build \
  --build-arg NEXT_PUBLIC_API_URL=$BACKEND_URL \
  -t us-central1-docker.pkg.dev/ecomm-support-prod/ecomm-support/frontend:latest \
  ./frontend-chat

# 4. Push + deploy frontend
docker push us-central1-docker.pkg.dev/ecomm-support-prod/ecomm-support/frontend:latest
gcloud run deploy ecomm-frontend \
  --image us-central1-docker.pkg.dev/ecomm-support-prod/ecomm-support/frontend:latest \
  --platform managed --region us-central1 \
  --allow-unauthenticated \
  --memory 512Mi

What needs to change in your backend code

Needs change

src/sql_agent/core/generator.py
Needs change
Uses sqlite3.connect(self.db_path) with a hardcoded default path "ecommerce.db". Replace with psycopg2 reading DATABASE_URL from env.
src/sql_agent/core/validator.py
Needs change
Uses sqlite3.connect(self.db_path) and pd.read_sql_query() against SQLite. Replace connection with psycopg2 + DATABASE_URL.
requirements.txt
Add packages
Add psycopg2-binary and cloud-sql-python-connector[pg8000] for Cloud SQL Auth Proxy support.
A
Cloud SQL
Create Cloud SQL instance + database
Create the instance in us-east4 matching your config, create the database, and set the root password.
bash
# Create PostgreSQL 17 instance
gcloud sql instances create ecomm-support-db \
  --database-version=POSTGRES_17 \
  --region=us-east4 \
  --tier=db-custom-2-8192 \
  --storage-size=20GB \
  --storage-type=SSD \
  --no-storage-auto-increase

# Create database + set password
gcloud sql databases create ecommerce --instance=ecomm-support-db
gcloud sql users set-password postgres \
  --instance=ecomm-support-db \
  --password=YOUR_STRONG_PASSWORD
B
Secret Manager
Store database URL in Secret Manager
Never hardcode credentials. Store the full connection string as a secret.
bash
# Get instance connection name
INSTANCE_CONNECTION=$(gcloud sql instances describe ecomm-support-db \
  --format='value(connectionName)')
# e.g. ecomm-support-prod:us-east4:ecomm-support-db

# Store DB URL (uses Cloud SQL socket path — no public IP needed)
echo -n "postgresql+psycopg2://postgres:YOUR_PASSWORD@/ecommerce?host=/cloudsql/$INSTANCE_CONNECTION" | \
  gcloud secrets create DATABASE_URL --data-file=-

# Grant Cloud Run access to the secret
gcloud secrets add-iam-policy-binding DATABASE_URL \
  --member="serviceAccount:PROJECT_NUMBER-compute@developer.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"
C
IAM
Grant Cloud Run access to Cloud SQL
Cloud Run connects via the built-in Cloud SQL Auth Proxy. Just grant the IAM role.
bash
gcloud projects add-iam-policy-binding ecomm-support-prod \
  --member="serviceAccount:PROJECT_NUMBER-compute@developer.gserviceaccount.com" \
  --role="roles/cloudsql.client"
D
Cloud Run
Deploy backend with Cloud SQL attached
Add --add-cloudsql-instances flag — Cloud Run handles the proxy socket automatically. Inject both secrets.
bash
INSTANCE_CONNECTION=$(gcloud sql instances describe ecomm-support-db \
  --format='value(connectionName)')

gcloud run deploy ecomm-backend \
  --image us-central1-docker.pkg.dev/ecomm-support-prod/ecomm-support/backend:latest \
  --platform managed --region us-central1 \
  --allow-unauthenticated \
  --memory 2Gi --cpu 2 --timeout 120 \
  --add-cloudsql-instances $INSTANCE_CONNECTION \
  --set-secrets GOOGLE_API_KEY=GOOGLE_API_KEY:latest,DATABASE_URL=DATABASE_URL:latest \
  --set-env-vars BOT_DOMAIN=ecommerce
E
Migration
Migrate SQLite data → PostgreSQL
One-time migration of your existing ecommerce.db into Cloud SQL using pgloader via Docker.
bash
# Enable public IP temporarily for migration (disable after)
gcloud sql instances patch ecomm-support-db --assign-ip

# Run pgloader via Docker (works on Windows/Mac/Linux)
docker run --rm -v ${PWD}/backend:/data dimitri/pgloader:latest \
  pgloader sqlite:///data/ecommerce.db \
  "postgresql://postgres:YOUR_PASSWORD@CLOUD_SQL_PUBLIC_IP/ecommerce"

# Remove public IP after migration
gcloud sql instances patch ecomm-support-db --no-assign-ip

Key decisions to make
🌐 Do you need a custom domain?
Yes → Add Cloud Load Balancer + SSL cert + domain mapping. I'll write those steps when ready.
No → Cloud Run gives you a free *.run.app HTTPS URL automatically. No extra setup needed.
⚡ Demo or production traffic?
Production → Migrate SQLite to Cloud SQL (PostgreSQL). Required for concurrent writes and reliability.
Demo / Internal → SQLite mounted from Cloud Storage is fine. Zero extra cost or infrastructure.
